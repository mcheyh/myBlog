## 设计过程

### 步骤一：确定设计方向与布局

- **操作内容**：规划个人主页的整体结构——采用毛玻璃卡片风格的单页布局，包含「关于我」、「联系方式」、「教育经历」、「竞赛科研」、「作品展示」五个板块，顶部设置粘性导航栏。
- **使用的工具**：Claude
- **具体做法**：向Claude描述需求：「设计一个个人主页，使用毛玻璃效果，包含自我介绍、联系方式、教育经历、竞赛科研、B站作品展示等板块，背景使用个人照片」。Claude生成了完整的HTML+CSS代码框架，包含Tailwind风格的原子化CSS类和毛玻璃卡片效果。
- **结果**：获得了结构清晰、视觉美观的页面框架，包含完整的导航系统和五个内容板块。

### 步骤二：填充个人信息与数据

- **操作内容**：将个人真实信息（联系方式、教育经历、竞赛获奖等）编写为JavaScript数据文件，实现数据与页面分离。
- **使用的工具**：Claude
- **具体做法**：分别创建 `about.js`、`contact.js`、`education.js`、`contest.js` 等数据文件，每个文件暴露对应的数组或对象供页面动态渲染。Claude帮助设计了数据结构并生成渲染逻辑。
- **结果**：页面内容通过JavaScript动态生成，修改个人信息只需编辑对应的数据文件，维护方便。

### 步骤三：对接B站API实现作品展示

- **操作内容**：通过B站API获取个人视频列表，动态展示在「作品展示」板块。
- **使用的工具**：Claude
- **具体做法**：Claude帮助编写了WBI签名算法（`bilibili.js`），实现了对B站API的合法调用。同时配置了`works.js`作为静态数据兜底，当API请求失败时自动切换为预置数据，保证页面始终有内容展示。
- **结果**：页面加载时自动请求B站API获取最新视频，点击视频卡片弹出模态播放器进行播放，用户体验流畅。

### 步骤四：解密WPK模型文件获取Live2D素材

- **操作内容**：将加密的Live2D模型文件（`小黑.wpk`）解密，提取出模型所需的 `.moc3`、纹理贴图、物理配置文件等。
- **使用的工具**：Claude + wpk解密工具
- **具体做法**：在Claude的指导下，定位WPK文件的加密方式，使用对应工具解密得到模型原始文件：`Moc_0.moc3`（模型网格）、`Textures_0_0.png`（纹理贴图，1.8MB）、`Physics_0.json`（物理动画参数）、`model0.json`（模型配置）。
- **结果**：成功获得小黑猫Live2D模型的完整素材文件，为后续集成看板娘做好准备。

### 步骤五：集成Live2D看板娘

- **操作内容**：将小黑猫Live2D模型作为看板娘嵌入网页左下角，支持鼠标追踪视线、点击触发反馈对话。
- **使用的工具**：Claude
- **具体做法**：
  1. Claude选择了 `pixi-live2d-display`（Cubism 4专版）+ `PixiJS v6` 作为渲染方案
  2. 采用了 `live2dcubismcore@1.0.2` 包中的 **asm.js版本Cubism Core**（纯JavaScript实现，无需WebAssembly文件）
  3. 编写 `live2d.js` 实现画布创建、模型加载、鼠标追踪平滑插值、点击交互与消息气泡
  4. 实现响应式设计：根据屏幕宽度（桌面/平板/手机）自动调整看板娘尺寸和位置
- **结果**：小黑猫成功显示在页面左下角，眼睛和头部会跟随鼠标移动，点击会触发互动对话。

### 步骤六：解决Chrome file://协议兼容性问题（关键步骤）

- **操作内容**：解决双击HTML文件打开时，Chrome浏览器因安全策略阻止加载本地文件的问题。
- **使用的工具**：Claude
- **具体做法**：该阶段经历了多轮排查和方案迭代：
  1. **第一轮**：使用CDN加载库文件 → 报错「Cubism 2 runtime missing」
  2. **第二轮**：切换到Cubism 4专版pixi-live2d-display + 本地库文件 → 报错「XHR blocked by CORS」
  3. **第三轮**：将模型JSON和.moc3文件编码为base64 data URL嵌入JavaScript → 报错「贴图CORS blocked」
  4. **最终方案**：将所有依赖的外部文件（.moc3、.json、.png）全部以base64 data URL形式内嵌到JavaScript代码中，实现 **零外部文件请求**。`live2d-data.js`（213KB）包含模型数据和物理参数，`live2d-texture.js`（2.4MB）包含贴图数据。
- **结果**：彻底解决了Chrome对file://协议的安全限制问题，实现了「双击HTML即可浏览」的作业要求。整个Live2D系统完全自包含，无任何CDN或外部文件依赖。

### 步骤七：优化看板娘外观与交互

## 遇到的问题与解决方案

### 问题一：WPK加密文件无法直接使用

- **问题现象**：从网络获取的Live2D模型文件为 `.wpk` 加密格式，无法被网页直接加载。
- **解决过程**：在Claude的帮助下分析WPK文件格式（实为ZIP加密压缩包），使用解密工具提取出内部的 `.moc3` 模型文件、纹理贴图和物理配置文件。
- **最终结果**：成功获得完整的Live2D模型素材，所有文件存放于 `live2d/` 目录。

### 问题二：Chrome浏览器阻止file://协议下的文件加载（核心难题）

- **问题现象**：双击 `index.html` 在Chrome中打开后，Live2D看板娘完全不显示。控制台报错：
  - `Access to XMLHttpRequest at 'file:///...' blocked by CORS policy`
  - `Access to image at 'file:///...' blocked by CORS policy`
- **解决过程**：这是整个作业中耗时最长的环节，经历了四次方案迭代：
  1. **方案A（CDN加载）**：通过CDN加载Cubism Core和PixiJS库 → 失败，官方Cubism Core的WASM文件加载依赖fetch()，而fetch()在file://下被Chrome阻止
  2. **方案B（本地库 + asm.js Core）**：使用纯JavaScript版Cubism Core（无需WASM）→ 部分成功，但模型文件（.json、.moc3、.png）的XHR请求仍被CORS阻止
  3. **方案C（data URL内嵌模型）**：将模型JSON和.moc3编码为base64 data URL → 部分成功，但纹理贴图仍通过Image元素加载，Chrome将file://图片视为跨域，阻止WebGL纹理上传
  4. **方案D（全量内嵌）**：将 **所有** 外部资源（模型JSON、.moc3文件、物理JSON、纹理PNG）全部以base64 data URL形式内嵌到JavaScript文件中。data URL由浏览器内存直接处理，不涉及file://请求 → **彻底成功**
- **最终结果**：Live2D看板娘能够在Chrome中正常显示和交互，完美满足「双击HTML即可浏览」的作业要求。

### 问题三：pixi-live2d-display版本兼容性问题

- **问题现象**：初始使用的pixi-live2d-display版本报错 `Cannot find Cubism 2 runtime`，因为它只支持Cubism 2.1模型，而小黑的模型是Cubism 3格式（.moc3）。
- **解决过程**：在Claude帮助下识别出版本问题——换用Cubism 4专版 (`cubism4.min.js`)，该版本专为Cubism 3/4模型设计，不依赖Cubism 2.1运行时。同时发现配套的PixiJS也是v5.3.6（过旧），需升级到v6.5.2以匹配pixi-live2d-display的API要求。
- **最终结果**：三个核心库（PixiJS v6.5.2 + Cubism 4专版pixi-live2d-display + asm.js Cubism Core）版本正确匹配，模型成功渲染。

### 问题四：贴图路径重复拼接

- **问题现象**：模型加载时报错贴图路径为 `live2d/live2d/Textures_0_0.png`，路径重复导致文件找不到。
- **解决过程**：排查发现是pixi-live2d-display的路径解析机制导致——传入的settings对象需要 `url` 字段作为基准路径，插件会将 `FileReferences.Textures` 中的相对路径与此基准拼接。原贴图路径写了完整相对路径 `live2d/Textures_0_0.png`，与基准 `live2d/model0.json` 拼接后变成 `live2d/live2d/Textures_0_0.png`。
- **最终结果**：将贴图路径改为 `Textures_0_0.png`（仅文件名），正确解析为 `live2d/Textures_0_0.png`。

## 总结与反思

- 本次作业中，我学到了：
  - HTML/CSS网页设计的基础知识，包括毛玻璃效果（backdrop-filter）、CSS Grid/Flexbox布局、响应式设计等
  - Live2D Cubism模型的渲染原理——从模型文件结构（.moc3网格、纹理贴图、物理引擎参数）到Web端渲染管线（Cubism Core → PIXI.js → WebGL）
  - 浏览器安全策略（CORS、file://协议限制）的基本原理和应对方案
  - 前端工程化的基本实践——文件结构组织、Git版本控制、数据与视图分离

- AI工具对我的帮助主要体现在：
  - **代码生成效率**：原本需要手写数百行的HTML/CSS/JS代码，在AI辅助下几轮对话即可完成框架搭建
  - **深度问题排查**：Chrome file:// CORS问题涉及浏览器安全策略、WASM加载、WebGL纹理上传等多个底层机制，AI帮助我逐层分析调用链，定位根本原因
  - **方案迭代优化**：在四次方案迭代中，AI能够快速提出替代方案并实现验证，大大加速了试错过程
  - **知识扩展**：通过AI的详细解释，我接触并理解了Live2D SDK、Emscripten asm.js/WASM编译、base64 data URL等较为深入的技术概念

- 我认为AI工具的局限性在于：
  - AI对特定领域知识的准确性有限——例如初始建议的CDN方案在实际运行中并不稳定，需要通过实际测试来验证
  - AI无法直接操作浏览器进行实时调试，某些问题（如file:// CORS）需要用户反馈具体报错信息后才能精准定位
  - 对大型二进制文件的处理存在限制——例如纹理贴图（1.8MB）的base64编码和分析需要多轮交互
  - AI的方案选择有时会陷入局部最优，需要人工判断整体方向（例如本报告中从CDN方案转向全量内嵌方案的决策）

- 如果重新做一次，我会：
  - 一开始就考虑到「双击HTML即可浏览」意味着必须兼容file://协议，提前规划好全离线方案
  - 在技术选型时优先选择兼容性更好的方案（如asm.js代替WASM），避免多次方案切换
  - 更早进行文件结构整理，避免后期重构路径引用
  - 尝试加入更多个性化元素，如背景音乐、页面转场动画等，丰富多媒体元素种类
