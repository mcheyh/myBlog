### 步骤七：解决Chrome file://协议兼容性问题（关键步骤）

- **操作内容**：解决双击HTML文件在Chrome中打开时，浏览器因安全策略阻止加载本地文件（CORS错误）导致Live2D无法显示的问题。
- **使用的工具**：Claude + DeepSeek
- **具体做法**：该阶段经历了四轮排查和方案迭代，是整个作业中耗时最长的环节：
  1. **第一轮（CDN方案，Claude）**：尝试通过CDN加载Cubism Core和PixiJS库 → 失败，官方Cubism Core的WASM文件加载依赖fetch()，Chrome在file://协议下禁止fetch()跨域请求
  2. **第二轮（本地库+asm.js Core，DeepSeek + Claude）**：向DeepSeek咨询"Chrome file:// CORS WASM alternative"。DeepSeek深入分析了Chrome对file://协议的安全限制层级，指出不仅fetch()会被阻止，XMLHttpRequest和Image加载同样受限，建议使用asm.js版Cubism Core（纯JavaScript，不依赖WASM和fetch）。Claude据此切换到asm.js版本 → 模型核心加载成功，但模型文件（.json、.moc3、.png）的XHR请求仍被CORS阻止
  3. **第三轮（data URL内嵌模型文件，Claude）**：Claude将模型JSON和.moc3文件编码为base64 data URL嵌入JavaScript → 模型结构和网格加载成功，但纹理贴图通过Image元素加载时，Chrome将file://来源的图片视为跨域，阻止WebGL的 `texImage2D` 纹理上传调用
  4. **第四轮（全量内嵌 — 最终方案，DeepSeek + Claude）**：针对贴图CORS问题，DeepSeek解释了WebGL的安全模型——`texImage2D` 在处理跨域图片时要求图片设置了 `crossOrigin` 属性且服务器返回了正确的CORS头，而file://协议无法满足这个条件，因此唯一彻底的解决方案是将纹理图片也转为data URL。Claude据此将 **所有** 外部资源全部以base64 data URL内嵌：`live2d-data.js`（213KB，含模型JSON和物理参数）、`live2d-texture.js`（2.4MB，含纹理PNG图片）。data URL由浏览器内存直接处理，完全绕过了file://请求机制
- **结果**：彻底解决了Chrome对file://协议的安全限制，实现了「双击HTML即可浏览」的作业要求。整个Live2D系统完全自包含，零外部文件请求、零CDN依赖。

## 遇到的问题与解决方案

### 问题一：WPK加密文件无法直接使用

- **问题现象**：从网络获取的Live2D模型文件为 `.wpk` 加密格式（约2.2MB），无法被网页直接加载。
- **解决过程**：在DeepSeek中搜索WPK文件格式的相关资料，了解到WPK本质上是带密码保护的ZIP压缩包，并获取了已知的开源解密工具列表。随后在Claude的帮助下安装并运行解密工具，成功提取出内部的 `.moc3`、`.png` 和 `.json` 文件。
- **最终结果**：成功获得完整的Live2D模型素材，所有原始文件存放于 `live2d/` 目录。

### 问题二：Chrome浏览器阻止file://协议下的文件加载（核心难题）

- **问题现象**：双击 `index.html` 在Chrome中打开后，Live2D看板娘完全不显示。控制台报错：
  - `Access to XMLHttpRequest at 'file:///...' blocked by CORS policy`
  - `Access to image at 'file:///...' blocked by CORS policy`
- **解决过程**：这是整个作业中耗时最长的环节，共经历四次方案迭代：
  1. **方案A（CDN加载）** — Claude实施 → 失败，官方Cubism Core的WASM加载依赖fetch()，Chrome在file://下禁止fetch()
  2. **方案B（本地库 + asm.js Core）** — DeepSeek建议使用纯JavaScript版Cubism Core（asm.js编译产物，无需WASM），Claude实施 → 部分成功，但模型文件的XHR请求仍被CORS阻止
  3. **方案C（data URL内嵌模型文件）** — Claude将模型JSON和.moc3编码为base64 → 部分成功，但纹理贴图通过Image元素加载时触发WebGL跨域纹理上传限制
  4. **方案D（全量内嵌）** — DeepSeek解释了WebGL `texImage2D` 的跨域安全模型后，Claude将所有外部资源全部以base64 data URL嵌入JavaScript → **彻底成功**
- **最终结果**：Live2D看板娘能够在Chrome中通过双击HTML文件正常显示和交互，完美满足「双击HTML即可浏览」的作业要求。

### 问题三：pixi-live2d-display版本兼容性问题

- **问题现象**：初始使用的pixi-live2d-display版本报错 `Cannot find Cubism 2 runtime`，它只支持Cubism 2.1模型（.moc文件），而小黑模型是Cubism 3格式（.moc3文件）。
- **解决过程**：Claude帮助识别出版本不匹配问题，并查找确认了各版本对Cubism模型格式的支持情况——Cubism 2专版仅支持.moc，Cubism 4专版支持.moc3。最终换用Cubism 4专版（`cubism4.min.js`），同步将PixiJS从v5.3.6升级到v6.5.2以匹配pixi-live2d-display的API要求。
- **最终结果**：PixiJS v6.5.2 + Cubism 4专版pixi-live2d-display + asm.js Cubism Core，三个核心库版本正确匹配，模型成功渲染。

### 问题四：贴图路径重复拼接

- **问题现象**：模型加载时报错贴图路径为 `live2d/live2d/Textures_0_0.png`，路径重复导致文件找不到。
- **解决过程**：排查发现是pixi-live2d-display的路径解析机制导致——传入的settings对象以 `url` 字段为基准路径，插件将 `FileReferences.Textures` 中的相对路径与基准拼接。原贴图路径写了完整相对路径 `live2d/Textures_0_0.png`，与基准 `live2d/model0.json` 拼接后变成 `live2d/live2d/Textures_0_0.png`。
- **最终结果**：将贴图路径改为 `Textures_0_0.png`（仅文件名），正确解析为 `live2d/Textures_0_0.png`。

### 问题五：B站API WBI签名鉴权

- **问题现象**：直接调用B站API返回错误码，提示需要WBI签名验证。
- **解决过程**：向DeepSeek咨询"B站API WBI签名算法原理"，了解到WBI签名需先从nav接口获取 `wbi_img` 的img_key和sub_key，经mixinKeyEncTab置换得到mixin_key，再对排序后的请求参数拼接mixin_key后取MD5摘要。DeepSeek还提供了Python参考实现。Claude据此将算法翻译为纯JavaScript（含完整的MD5实现），并与JSONP跨域请求封装整合为 `bilibili.js`。
- **最终结果**：成功通过B站官方API获取视频列表数据，实现了动态作品展示功能。API失败时有静态数据兜底。

## 总结与反思

- 本次作业中，我学到了：
  - HTML/CSS网页设计的基础知识，包括毛玻璃效果（backdrop-filter）、CSS Grid/Flexbox布局、响应式设计等
  - Live2D Cubism模型的渲染原理——从模型文件结构（.moc3网格、纹理贴图、物理引擎参数）到Web端渲染管线（Cubism Core → PIXI.js → WebGL）
  - 浏览器安全策略（CORS、file://协议限制、WebGL跨域纹理上传）的基本原理和应对方案
  - 前端工程化的基本实践——文件结构组织、Git版本控制、数据与视图分离
  - **如何有效利用多个AI工具分工协作完成一个完整项目**——不同AI工具有各自的优势领域，合理分工可以发挥各自所长

- AI工具对我的帮助主要体现在：
  - **代码实现能力（Claude）**：原本需要手写数百行的HTML/CSS/JS代码，在Claude的辅助下几轮对话即可完成框架搭建。尤其是Live2D整合这种涉及多个第三方库和底层WebGL的技术栈，Claude的代码生成和调试能力极大降低了实现门槛
  - **内容创作能力（ChatGPT）**：ChatGPT在中文文案撰写方面表现出色，生成的自我介绍和爱好描述自然流畅、富有个人特色，节省了大量构思和反复修改的时间。在视觉设计方面，ChatGPT作为"设计评审"角色提供的配色和排版建议也非常实用
  - **技术攻坚能力（DeepSeek）**：在WPK解密和file:// CORS这两个关键难题上，DeepSeek展现了深厚的技术分析能力。它不仅能回答"怎么做"，还能解释"为什么"——例如对Chrome安全策略的逐层分析和WebGL跨域纹理上传机制的解读，帮助我真正理解了问题的本质，而不仅仅是套用解决方案
  - **协同互补效应**：三个AI工具在各自的优势领域独立工作，又在关键节点形成交叉。例如在CORS问题攻关中，DeepSeek负责分析问题本质和方案论证，Claude负责将方案落地为代码，两者配合使得从"发现问题"到"彻底解决"的路径大大缩短

- 不同AI工具的优势定位：
  - **Claude（Claude Code）**：在代码生成、系统整合和调试方面能力最为突出。适合作为项目的"主工程师"，负责从设计稿到代码的转化、多技术栈的整合、以及持续的调试迭代。IDE集成模式让代码修改和迭代效率极高
  - **ChatGPT**：在中文内容生成、创意设计和语言表达方面表现最佳。适合作为项目的"内容总监"和"设计顾问"，负责文案撰写、视觉审阅和报告润色
  - **DeepSeek**：在技术深度问题和国内技术生态方面有明显优势。适合作为项目的"技术顾问"，在遇到未知领域或复杂技术难题时提供深入分析和方案论证。尤其是对于B站API、WPK格式等国内开发者常见的技术话题，DeepSeek的知识储备更加对口

- 我认为AI工具的局限性在于：
  - AI对特定领域知识的准确性有限——初始建议的CDN方案在实际file://环境中并不稳定，需要通过实际测试来验证；ChatGPT偶尔会给出看似合理但在浏览器安全沙箱中无法实施的CSS建议
  - AI无法直接操作浏览器进行实时调试，某些问题（如file:// CORS）需要用户反馈具体的控制台报错信息后才能精准定位
  - 不同AI工具给出的建议可能相互矛盾，需要人工判断和取舍——例如在Cubism Core的选择上，一个工具倾向官方SDK，另一个建议社区方案，最终决策依赖于对作业约束条件（必须兼容file://）的理解
  - 对大型二进制文件的处理存在限制——纹理贴图（1.8MB）的base64编码和嵌入需要多轮交互和人工验证
  - AI的方案选择有时会陷入局部最优，需要人工判断整体方向——例如从CDN转向全量内嵌的架构决策，是基于对"双击HTML即可浏览"这一硬性约束的全局判断

- 如果重新做一次，我会：
  - 在项目启动时就明确三个AI工具的分工定位：ChatGPT负责内容与设计 → DeepSeek负责技术方案论证 → Claude负责代码实现，让协作流程更加有序高效
  - 一开始就考虑到「双击HTML即可浏览」意味着必须兼容file://协议，提前用DeepSeek做技术可行性调研，将全离线方案作为初始架构，避免四次方案迭代的试错成本
  - 在技术选型时优先选择兼容性更好的方案（如asm.js代替WASM），将DeepSeek的底层分析环节前置到方案设计阶段而非问题排查阶段
  - 更早进行文件结构整理，避免后期因路径变更导致的批量重构
  - 尝试加入更多个性化元素，如用AI音乐工具（Suno）生成一段背景音乐、添加页面滚动视差效果等，进一步丰富多媒体元素种类
